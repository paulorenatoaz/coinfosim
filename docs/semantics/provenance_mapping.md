# Provenance Mapping — PROV-O Roles

This document defines the initial mapping between CoInfoSim objects and
[W3C PROV-O](https://www.w3.org/TR/prov-o/) roles. It is provenance-ready
metadata, not a complete domain ontology and not an RDF/OWL runtime
dependency: no reasoner or triple store is required to produce or consume
the JSON-LD emitted by `src/coinfosim/provenance/`.

## Role table

| CoInfoSim object | PROV-O role |
|---|---|
| Recovered `gh-pages` result-data file (`result_data_*.json.gz`) | `prov:Entity` |
| Dataset snapshot | `prov:Entity` |
| Fixed train/test split | `prov:Entity` |
| Preprocessing metadata | `prov:Entity` |
| Generator fit | `prov:Activity` |
| Fitted generator | `prov:Entity` |
| Monte Carlo simulation run (original) | `prov:Activity` |
| Persisted simulation result | `prov:Entity` |
| Predictive cooperation profile recomputation | `prov:Activity` |
| Predictive cooperation profile | `prov:Entity` |
| HTML/PDF report regeneration | `prov:Activity` |
| Report artifact | `prov:Entity` |
| CoInfoSim commit/version | `prov:SoftwareAgent` (via `prov:wasAssociatedWith`) |

## Relations used

- `prov:used` — an activity used an entity as input (e.g. the profile
  recomputation activity used the recovered `result_data_*.json.gz` entity).
- `prov:wasGeneratedBy` — an entity was produced by an activity (e.g. the
  predictive cooperation profile entity was generated by the recomputation
  activity).
- `prov:wasDerivedFrom` — an entity was derived from another entity (e.g. the
  regenerated report artifact was derived from the recovered result-data
  entity and the recovery commit).
- `prov:wasAssociatedWith` — an activity was associated with an agent (e.g.
  the report-regeneration activity was associated with the current
  CoInfoSim commit).

## Commit distinction

Provenance records distinguish, and never collapse into a single "scientific
commit":

1. the original simulation commit, when it is persisted in recovered
   metadata;
2. the `gh-pages` publication commit (the recovery source);
3. the current semantic-refactor commit;
4. the report-regeneration commit (which may equal 3, or be later).

## Source vs. derived entities

Recovered `gh-pages` artifacts are immutable **source** entities. Every new
artifact produced by this refactor records a `prov:wasDerivedFrom` relation
to:

- the exact `result_data` file hash(es) it was recomputed from;
- the scenario/simulation metadata it read;
- the recovery commit SHA (`origin/gh-pages` at recovery time).

## Path and determinism rules

- Persist repository-relative paths only. Never persist `/home/...`,
  usernames, or other machine-specific/temporary worktree paths.
- Use SHA-256 for all artifact hashes.
- Serialize with sorted keys and stable ordering.
- A generated timestamp may appear as metadata but is never the sole
  identity of an artifact — identity is derived from run IDs, artifact
  hashes, and relative paths.
